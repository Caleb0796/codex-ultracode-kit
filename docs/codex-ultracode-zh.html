01核心洞察：为什么不能只靠提示词

这是整套方案的前提，也是 Codex 与 Claude 最根本的差别。

Claude Code 的「几十个智能体」并非模型自己决定生成的，而是来自一个确定性脚本（Workflow 工具）——由外层程序用 parallel() / pipeline() 循环决定并发数量。Codex 的原生 spawn_agent 恰好相反：由模型驱动，且被训练得很保守。官方文档明确写着「Codex 仅在用户明确要求时才生成智能体」，典型示例只生成 2–6 个。

结论：无论怎么写提示词，都无法稳定让 Codex 模型生成几十个智能体——少生成是「设计如此」，不是 bug。要逼近 Claude 的规模，必须把并发数量握在外层脚本手里，用 codex exec 作为单个智能体的原语。这就是 codex_workflow.py 的全部思路。

03触发方式与门控

Codex 的「明确要求才生成」是一道硬门控；关键词就是用来满足它的。

触发词
消息中出现独立单词 ultra（非 ultrasound 之类的子串）或 ultracode → 视为「明确要求并行子智能体」，技能启动。
自然语言
不用关键词、直接说「用并行子智能体…」「派一个 skeptic 去验证…」同样满足原生门控。
默认行为
两者都没有 → 单智能体独立完成（成本控制保证：无关键词不扇出）。
已修正的坑：技能描述最初会在「做个全面 review」这类无关键词请求上自我授权扇出，与原生门控冲突。已收紧为仅在关键词或明确的多智能体请求时触发。

04三层架构：按规模选工具

不是一个旋钮，而是三条路径，并发能力依次递增。

层级工具并发上限适用
会话内 · 模型协调spawn_agent（Ultracode 技能）受 max_threads 限制（默认 6 → 已设 16）2–6 个异构子任务，模型自己协调与综合
会话内 · 同质批量spawn_agents_on_csvmin(请求值, 64, max_threads)「对这 N 个文件各做 X」——数据驱动，绕过模型保守性
外部 · 确定性codex_workflow.py你设定的 N（受 API 限流 / 内存约束）真正的数十个智能体，确定性扇出
关键事实：spawn_agents_on_csv 的「16 并发」会被 max_threads 卡住——默认 max_threads=6 时，即使 CSV 请求 64 也只跑 6 个。所以提升 max_threads 是前提。

05质量模式（Ultracode 的「方法论」）

引擎只是并发；让它成为「ultracode」而非「并行 map」的，是这些模式。已在 codex_patterns.py 中实现。

对抗式验证 adversarial_verify

对每条结论生成 N 个 skeptic，各自从不同视角（正确性 / 安全 / 可复现）尝试推翻，多数通过才保留。自己给自己打分是表演——独立性才是关键。

评审团 judge_panel

从不同角度独立产出 N 个方案，再由裁判智能体打分、择优综合。解空间宽时优于「一稿反复改」。

循环到枯竭 loop_until_dry

对未知规模的发现（bug、边界情形）反复跑，跨轮去重，连续若干轮无新增才停。固定次数会漏掉长尾。

完整性批判 completeness_critic

结束前重读原始需求，把每条明确要求映射到对应产出——未映射即「未完成」。修的是检索遗漏，不是自我评价。

两段式评审 two-stage review

写码单元先查「规格符合」（漏了什么 / 多做了什么 / 误解了什么），再查代码质量。规格不过不进质量评审。

波次 + 槽位回收

单元多于并发上限时分波处理：生成到上限 → 等待 → 读结果并立即 close_agent 回收槽位 → 下一波。

06配置与上限（已对照 Rust 源码核实）

max_threads
默认 6；唯一校验是「≥1」，无上限钳制——可设 16/32/64，逐字生效（仅限 V1 多智能体路径，本机即是）。这是会话内树范围的并发存活上限。
CSV 硬上限
spawn_agents_on_csv 每作业最多 64 并发（MAX_AGENT_JOB_CONCURRENCY），且再被 max_threads 钳制。超过 64 无意义。
槽位泄漏
已完成但未 close_agent 的智能体仍占用槽位，长程波次不回收会中途撞限。
multi_agent_v2
处于开发中且关闭。不要开启——它会拒绝 max_threads 并把上限降到约 3。
实际瓶颈
账号级 API 限流（TPM/RPM）与本机内存，远早于任何配置数字。官方默认正是从 64 降到 16 以避免限流。

07环境约束（两个工具都改不了）

云策略强制审批：本账号的云端策略把审批强制为 OnRequest，-a never 被静默降级。后果——编排出的智能体能改文件（沙箱内写入无需审批），但不能跑 shell 命令（测试/构建在 exec 模式下被自动拒绝："command execution approval is not supported in exec mode"）。

因此写码智能体被要求只改文件，而验证（跑测试）必须放到编排器里、在收集 diff 之后执行——这本来也是更合理的分工。另外 worktree 内 git add -A 会带进无关的未跟踪文件（如 SCM 日志），应用 diff 时要限定到目标路径。

08Codex vs Claude Ultracode 差距对照

逐项对照（由两个对抗审计智能体核实）。引擎与方法论已对齐；三项无法复刻的属于 Claude Code 宿主层能力，Codex 的 codex exec 不暴露。

对齐 语义一致 已补齐 经审计后补上 部分 近似/弱化 有意偏离 设计取舍 无法复刻 宿主层能力
能力状态说明
agent / parallel / pipeline对齐屏障型 parallel（失败→None）、无屏障 pipeline（逐项链、(prev,item,index) 阶段签名、抛错→None）语义一致。
schema 结构化输出对齐Codex --output-schema 服务端强制；编排器再加 jsonschema 校验 + 不符重试。需 additionalProperties:false（已自动注入）。
质量模式（验证/评审团/枯竭/批判）已补齐codex_patterns.py 完整移植——这是「ultracode」与「并行 map」的分水岭。
worktree 隔离（并行写者）已补齐每个写者独立 worktree，返回 diff；重试不泄漏、清理 worktree+分支。已实测：2 写者互不冲突。
budget 令牌上限已补齐CODEX_WF_BUDGET 超限即 BudgetExceeded（原本只是只读计量）。
log() 进度叙述部分有 log() 平铺输出到 stderr；无 phase() 分组、无实时进度树。
role / agentType部分会话内有 explorer/worker/skeptic 角色；外部编排器以提示词框定角色（codex exec 无子智能体角色选择）。
MCP 工具访问有意偏离Workflow 智能体可用会话 MCP；本编排器默认 -c mcp_servers={} 跑「干净」环境以求快与确定。注：该参数只能部分抑制已配置服务器（仍会尝试连接）。
并发模型有意偏离Claude：min(16, 核数-2) 由宿主调度，模型不介入。Codex 会话内：模型驱动 + max_threads 上限（默认 6）。外部编排器：由你设定，与模型脱钩。
resume / 缓存（resumeFromRunId）无法复刻每次重跑全部智能体，无「未变前缀复用缓存」。长程迭代最大的缺口。变通：自建 prompt 内容哈希磁盘缓存（仅只读智能体）。
后台执行 + 实时进度无法复刻Workflow 立即返回 runId、后台运行、/workflows 实时树、完成通知。Python 脚本是阻塞的；最接近的是 nohup … & + tail log()。
嵌套 workflow() / args / 生命周期上限无法复刻Workflow 有内联子工作流、JSON args、1000 总量 / 4096 单批上限。Python 侧用普通函数与 argv 替代（非阻塞缺口），但无运行时上限保护。
一句话总结：引擎（agent/parallel/pipeline/schema）很相似，方法论（质量模式）已补齐；真正补不上的是 resume 缓存、实时后台进度 UI、智能体级 MCP——它们来自 Claude Code 的宿主，而非 codex exec 能提供的东西。

09验证记录：「看起来对」≠「真的对」

本方案每一项「能用」都经过实跑验证。以下是被测试揪出、随后修复的真实缺陷。

  • 管道缓冲死锁：编排器在 2 个智能体时通过、8 个时全部超时。对照实验证明 Codex 本身 8 并发正常（约 22s），缺陷在我的代码——capture_output=True 用管道，codex exec 输出量大塞满 ~64KB 管道缓冲导致 subprocess.run 死锁。修复：子进程输出重定向到文件，绝不用管道。
  • schema 假校验：原本只用贪婪正则 {.*} 取 JSON、不做任何校验，错误形状会静默通过。修复：平衡括号解析 + jsonschema 校验 + 不符重试。
  • worktree 泄漏 + 写者挂起：并行写者测试挂起并生成 4 个 worktree（2 智能体 × 2 重试）。根因是 exec 模式拒绝命令审批 + 重试路径泄漏 worktree/分支。修复：写者「只改文件」、重试前先清理、退出时连分支一起删——重测通过。
  • 六智能体硬上限：原生 spawn_agent 撞到 AgentLimitReached。根因是 max_threads 默认 6。修复并实测：-c agents.max_threads=16 下 8 个子智能体全部完成、无报错，随后写入配置。
诚实的保留：提升 max_threads 解除了硬上限，但模型仍被训练得保守——16 是上限不是承诺。要可靠的大规模扇出，外部 codex_workflow.py（数量由你定）仍是更可靠的路径。
